Originally a cloud-only feature (Atlas), vector search is now available in public preview for Community Edition (self-hosted).

Data Model: Adds a specific vector data type.

Quick Comparison

Building an ingestion pipeline for your home lab is a great way to handle your Markdown notes. Using Python with pymongo and a local embedding library like sentence-transformers allows you to keep everything self-hosted.

Here is a streamlined script to get you started.

The Hybrid Ingestion Script (Python)
This script will:

Read every .md file in a folder.

Clean the Markdown (removes basic frontmatter).

Embed the text into a 768-dimension vector (using a local model).

Upsert into MongoDB so you don't create duplicates if you run it twice.

Python
import os
import re
from pymongo import MongoClient, UpdateOne
from sentence_transformers import SentenceTransformer

# 1. Setup - Using a high-quality local model (no API key needed)
model = SentenceTransformer('all-mpnet-base-v2') 
client = MongoClient("mongodb://localhost:27017/")
db = client["ai_lab"]
collection = db["markdown_notes"]

def clean_markdown(text):
    # Optional: Strip YAML frontmatter if your notes have it
    return re.sub(r'^---.*?---', '', text, flags=re.DOTALL).strip()

def ingest_directory(path):
    operations = []
    
    for filename in os.listdir(path):
        if filename.endswith(".md"):
            with open(os.path.join(path, filename), 'r', encoding='utf-8') as f:
                content = f.read()
                cleaned_content = clean_markdown(content)
                
                # Generate local embedding
                vector = model.encode(cleaned_content).tolist()
                
                # Prepare document
                doc = {
                    "filename": filename,
                    "content": cleaned_content,
                    "embedding": vector,
                    "metadata": {
                        "source": "home_lab",
                        "char_count": len(cleaned_content)
                    }
                }
                
                # Use filename as the unique key to prevent duplicates
                operations.append(
                    UpdateOne({"filename": filename}, {"$set": doc}, upsert=True)
                )

    if operations:
        result = collection.bulk_write(operations)
        print(f"Ingested {result.upserted_count} new and updated {result.modified_count} existing notes.")

# Run it
ingest_directory("./my_notes")

This "voting" concept is essentially a Reinforcement Learning from Human Feedback (RLHF) loop for your knowledge base. In an Enterprise context, this is the difference between a system that just "retrieves" and one that "learns."

Implementing a memory-like weighting system solves the "garbage in, garbage out" problem of large-scale RAG (Retrieval-Augmented Generation).

1. The "Memory Weighting" Strategy
The system you heard about is often called Usage-Based Ranking or Dynamic Relevance Scoring. Here is how you can architect it in MongoDB:

The Counter System: Add a utility_score (integer) and a last_accessed (timestamp) to every document.

The Feedback Loop: When the AI uses a chunk to answer a question, you ask the user (or the AI itself, acting as a judge) to verify the quality.

Success (+1): The information was "Correct and Helpful."

Failure (-1): The information was "Hallucination-prone" or "Irrelevant."

The Decay Factor: Just like human memory, old info should lose weight if not used. You can calculate a "Current Value" using a Half-Life Decay formula:

V=S×e 
−λt
 
(Where V is current value, S is the utility score, and t is the time since last use.)

2. Other Advanced Strategies for "Quality" Ingestion
To ensure your Enterprise knowledge base doesn't become a digital landfill, consider these three additional layers:

A. LLM-as-a-Judge (Pre-Ingestion Scoring)
Don't just ingest everything. Before a meeting transcript or Markdown note hits the database, run it through a small, fast model (like Claude 3 Haiku or GPT-4o-mini) with a specific rubric:

Clarity: Is this transcript coherent? (Score 1-5)

Actionability: Does it contain tasks or decisions? (Score 1-5)

Density: Is the information-to-noise ratio high?

Enterprise Tip: If a transcript scores below a 2/5, flag it for "Human Review" before it’s allowed to influence the AI's "Long-Term Memory."

B. Semantic Deduplication
In an Enterprise environment, you often have 10 versions of the same "Onboarding" document.

How it works: When ingesting a new note, do a vector search for the top 3 similar notes.

The Choice: If the similarity is >95%, don't add a new note—merge them or version them. This prevents the AI from getting "confused" by conflicting instructions from 2022 vs. 2024.

C. Graph-Relational Weighting (The "Knowledge Web")
Instead of treating chunks as isolated islands, track how they relate.

If a Markdown note on "Java Microservices" is frequently cited alongside a "Spring Boot Meeting Transcript," create a strong link between them in your metadata.

When the AI searches for one, it automatically gets a "boosted" score for the other.
